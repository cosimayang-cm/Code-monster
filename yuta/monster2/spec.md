# Feature Specification: 彈窗連鎖顯示機制 (Popup Response Chain)

**Status**: Draft
**Owner**: Yuta
**Created**: 2025-01-18
**Last Updated**: 2025-01-18

---

## Overview

當用戶重新打開 App 並進入主畫面後，系統需要依照優先順序檢查各種彈窗的顯示條件，並以連鎖方式逐一顯示符合條件的彈窗。

### Business Value

- 確保用戶不會錯過重要資訊（新手教學、公告、簽到等）
- 避免同時顯示多個彈窗造成用戶困擾
- 提供有序且可預測的彈窗體驗
- 系統可輕易擴展新的彈窗類型，維護成本低

### Target Users

- App 用戶：在進入主畫面時接收各類彈窗通知
- 開發團隊：可輕易新增或調整彈窗類型與優先順序

## User Scenarios & Testing

### Primary User Flow

**場景1：用戶首次開啟 App（完整流程）**

```
進入主畫面
    ↓
檢查：是否看過新手教學？ ─── N ──→ 顯示新手教學彈窗 → 用戶關閉 ─┐
    │ Y                                                      │
    ↓←──────────────────────────────────────────────────────←┘
檢查：是否看過插頁式廣告？ ─── N ──→ 顯示插頁式廣告彈窗 → 用戶關閉 ─┐
    │ Y                                                        │
    ↓←────────────────────────────────────────────────────────←┘
檢查：是否看過新功能公告？ ─── N ──→ 顯示新功能彈窗 → 用戶關閉 ─┐
    │ Y                                                      │
    ↓←──────────────────────────────────────────────────────←┘
檢查：今日是否已簽到？ ─── N ──→ 顯示簽到彈窗 → 用戶關閉 ─┐
    │ Y                                                  │
    ↓←──────────────────────────────────────────────────←┘
檢查：有猜多空結果產生？ ─── Y ──→ 顯示猜多空結果彈窗 → 用戶關閉 ─┐
    │ N                                                        │
    ↓←────────────────────────────────────────────────────────←┘
結束（進入主畫面正常使用）
```

**場景2：用戶已完成部分項目**

1. 用戶開啟 App（已看過新手教學和廣告）
2. 系統跳過「新手教學」和「插頁式廣告」
3. 系統檢查是否有新功能公告
4. 若有未看過的公告，顯示「新功能彈窗」
5. 用戶關閉彈窗後，繼續檢查後續條件

**場景3：用戶所有條件都已滿足**

1. 用戶開啟 App（所有彈窗都已看過，今日已簽到，無預測結果）
2. 系統依序檢查所有 5 個條件
3. 所有條件都不符合，不顯示任何彈窗
4. 用戶直接進入主畫面使用 App

**場景4：中間有多個彈窗需要顯示**

1. 用戶開啟 App（已看過新手教學，但未看過廣告和公告）
2. 系統跳過「新手教學」
3. 顯示「插頁式廣告彈窗」，用戶關閉
4. 顯示「新功能彈窗」，用戶關閉
5. 檢查簽到狀態，若今日已簽到則跳過
6. 檢查猜多空結果，若無結果則結束

### Edge Cases

| 情境 | 預期行為 |
|------|----------|
| 快速關閉彈窗 | 系統正確觸發下一個彈窗檢查，不會遺漏 |
| 用戶在彈窗顯示中離開 App | 重新進入時從頭開始檢查整個 Chain |
| 某彈窗的資料載入失敗 | 跳過該彈窗，繼續檢查下一個 |
| 沒有設定任何 Handler | 系統安全結束，不顯示任何彈窗 |
| 只有最後一個彈窗符合條件 | 前 4 個都跳過，只顯示第 5 個彈窗 |
| 所有彈窗都符合條件 | 依序顯示全部 5 個彈窗，每個關閉後才顯示下一個 |

## Functional Requirements

### Must Have

| ID | 需求 | 驗收條件 |
|----|------|----------|
| FR-1 | 同一時間只顯示一個彈窗 | 任何時刻畫面上最多只有一個彈窗存在 |
| FR-2 | 彈窗按優先順序依序檢查 | 嚴格按照順序（1→2→3→4→5）依序檢查條件 |
| FR-3 | 關閉彈窗後繼續檢查下一個 | 用戶關閉當前彈窗後，系統自動檢查下一個彈窗條件 |
| FR-4 | 條件不符時跳過該彈窗 | 若條件不符合，不顯示該彈窗，直接檢查下一個 |
| FR-5 | 所有條件都不符時結束 | 若所有彈窗條件都不符合，連鎖結束，不顯示任何彈窗 |
| FR-6 | Handler 需有 next 屬性 | 每個 Handler 都能指向下一個 Handler，形成鏈狀結構 |
| FR-7 | Handler 需能檢查自身條件 | 每個 Handler 能獨立判斷自己的顯示條件是否滿足 |
| FR-8 | Handler 需能處理彈窗顯示 | 每個 Handler 能執行顯示彈窗的動作 |
| FR-9 | Handler 需能傳遞給下一個 | 條件不符或彈窗關閉後，能將控制權傳遞給下一個 Handler |

### Should Have

| ID | 需求 | 驗收條件 |
|----|------|----------|
| FR-10 | 可擴展性 | 新增彈窗類型只需新增 Handler 並加入 Chain，不修改現有 Handler |
| FR-11 | Handler 可獨立測試 | 每個 Handler 可以單獨進行單元測試，不依賴其他 Handler |
| FR-12 | 優先順序可調整 | 透過調整 Chain 串接順序即可改變優先順序 |
| FR-13 | Manager 提供啟動入口 | Manager 提供單一方法啟動整個檢查流程 |
| FR-14 | Manager 負責組裝 Chain | Manager 負責將所有 Handler 依序串接起來 |

### Won't Have (This Release)

- 彈窗動畫效果
- 彈窗顯示時間統計
- A/B Testing 支援
- 遠端設定彈窗順序
- 彈窗的 UI 實作（本作業專注於 Chain of Responsibility 邏輯）

## Success Criteria

| 指標 | 目標 |
|------|------|
| 彈窗顯示正確率 | 100% 符合條件的彈窗都能正確顯示 |
| 順序正確率 | 100% 按照定義的優先順序（1→2→3→4→5）顯示 |
| 單一彈窗原則 | 100% 任何時刻最多只有一個彈窗 |
| 連鎖傳遞正確率 | 100% 關閉彈窗後正確觸發下一個檢查 |
| 擴展性驗證 | 新增一種彈窗類型只需新增一個 Handler，不修改現有 Handler |
| 測試覆蓋 | 每個 Handler 都有獨立的單元測試，涵蓋「顯示」和「跳過」兩種情境 |

## Key Entities

### 彈窗類型與優先順序

| 優先順序 | 彈窗類型 | 英文名稱 | 顯示條件 |
|:--------:|----------|----------|----------|
| 1 | 新手教學 | Tutorial | 用戶尚未看過新手教學 |
| 2 | 插頁式廣告 | Interstitial Ad | 用戶尚未看過此廣告 |
| 3 | 新功能公告 | New Feature | 用戶尚未看過新功能公告 |
| 4 | 每日簽到 | Daily Check-in | 用戶今日尚未簽到 |
| 5 | 猜多空結果 | Prediction Result | 有預測結果產生 |

### 核心概念

| 概念 | 說明 | 職責 |
|------|------|------|
| **Handler（處理器）** | 負責處理單一類型彈窗的物件 | 1. 檢查自身條件<br>2. 顯示彈窗（若條件符合）<br>3. 傳遞給下一個 Handler |
| **Chain（責任鏈）** | 多個 Handler 依優先順序串接形成的鏈 | 依序傳遞處理請求 |
| **Manager（管理器）** | 負責組裝和啟動 Chain 的物件 | 1. 組裝 Handler Chain<br>2. 提供啟動入口<br>3. 觸發第一個 Handler |

### Handler 行為規格

每個 Handler 必須能夠：

1. **檢查條件** (`shouldHandle`)
   - 輸入：用戶狀態資料
   - 輸出：布林值（是否應該顯示此彈窗）

2. **處理請求** (`handle`)
   - 若條件符合：顯示彈窗，彈窗關閉後呼叫下一個 Handler
   - 若條件不符：直接呼叫下一個 Handler

3. **設定下一個** (`next`)
   - 持有對下一個 Handler 的參考
   - 可為空（表示這是最後一個 Handler）

## Assumptions

1. **彈窗關閉機制**：假設所有彈窗都有統一的關閉回調機制，關閉時會通知 Handler
2. **狀態持久化**：假設用戶的「已看過」狀態會被持久化儲存（本作業不實作持久化）
3. **同步檢查**：假設條件檢查是同步進行的，不涉及複雜的非同步操作
4. **單一入口**：假設彈窗連鎖只會在進入主畫面時觸發一次
5. **順序固定**：本次實作中，彈窗優先順序是固定的，由 Manager 在組裝時決定
6. **Mock UI**：本作業專注於 Chain of Responsibility 邏輯，彈窗 UI 使用 Mock 或 Console 輸出模擬

## Design Pattern Reference

本功能將使用 **Chain of Responsibility（責任鏈）** 設計模式實作：

### 設計要求對照

| 作業要求 | 對應規格 |
|----------|----------|
| 1. 定義 Protocol | FR-6, FR-7, FR-8, FR-9 + Handler 行為規格 |
| 2. 實作各種 Handler | 5 種彈窗類型，各自獨立實作 |
| 3. 串接 Handler Chain | FR-14: Manager 負責組裝 |
| 4. 建立 Manager | FR-13, FR-14 |
| 5. 可擴展性 | FR-10: 新增 Handler 不修改現有程式碼 |

### Chain of Responsibility 核心原則

1. **解耦請求發送者與接收者**：Manager 不需要知道哪個 Handler 會處理請求
2. **動態組合**：可以在運行時改變 Chain 的組成和順序
3. **單一職責**：每個 Handler 只負責自己類型的彈窗
4. **開放封閉**：新增彈窗類型只需新增 Handler，不修改現有代碼
