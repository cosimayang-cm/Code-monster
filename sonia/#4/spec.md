# Feature Specification: RPG 道具系統

**Feature Branch**: `003-rpg-item-system`
**Created**: 2026-01-30
**Status**: Draft
**Input**: User description: "RPG 遊戲的道具/物品欄/背包系統"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 穿戴與卸下裝備 (Priority: P1)

玩家可以將背包中的裝備穿戴到角色的對應欄位上，穿戴後角色的能力數值會即時更新。玩家也可以卸下已穿戴的裝備，將其放回背包。

**Why this priority**: 這是裝備系統的核心功能，沒有穿戴/卸下機制，整個物品系統就沒有實際用途。

**Independent Test**: 可以透過建立一個角色、一件裝備，然後執行穿戴/卸下操作來獨立測試，驗證數值計算是否正確。

**Acceptance Scenarios**:

1. **Given** 角色有空的頭盔欄位且背包中有一件頭盔, **When** 玩家將頭盔穿戴到頭盔欄位, **Then** 頭盔從背包移除並出現在頭盔欄位，角色數值增加該頭盔的屬性值
2. **Given** 角色已穿戴一件頭盔, **When** 玩家卸下頭盔, **Then** 頭盔回到背包，角色數值減少該頭盔的屬性值
3. **Given** 角色已穿戴一件頭盔且背包中有另一件頭盔, **When** 玩家用新頭盔替換, **Then** 舊頭盔回到背包，新頭盔穿戴到欄位，數值正確更新
4. **Given** 玩家有一件鞋子, **When** 玩家嘗試將鞋子穿戴到頭盔欄位, **Then** 系統拒絕操作並顯示錯誤訊息

---

### User Story 2 - 物品生成與唯一識別 (Priority: P1)

系統可以根據物品模板生成物品實例，每個實例擁有全域唯一的識別碼。同一模板可以生成多個不同的實例。

**Why this priority**: 物品生成是系統運作的基礎，必須能正確建立物品才能進行後續的穿戴、背包管理等操作。

**Independent Test**: 可以透過從同一模板生成多個物品，驗證每個實例的 UUID 都不相同。

**Acceptance Scenarios**:

1. **Given** 存在「鐵製頭盔」模板, **When** 系統生成一個物品實例, **Then** 該實例擁有唯一的 UUID 且包含模板定義的所有基礎屬性
2. **Given** 存在「鐵製頭盔」模板, **When** 系統連續生成兩個實例, **Then** 兩個實例的 UUID 不同，但模板 ID 相同
3. **Given** 一個物品實例的 UUID, **When** 系統查詢該 UUID, **Then** 能快速找到對應的物品實例

---

### User Story 3 - 詞條系統與隨機生成 (Priority: P2)

裝備具有主詞條和副詞條。主詞條固定於物品上，副詞條根據稀有度隨機生成。詞條為裝備提供額外的屬性加成。

**Why this priority**: 詞條系統增加遊戲深度和隨機性，但需要先有基礎的物品和穿戴系統才能展現價值。

**Independent Test**: 可以透過生成不同稀有度的物品，驗證副詞條數量和隨機分布是否符合規則。

**Acceptance Scenarios**:

1. **Given** 一件傳說品質裝備, **When** 生成該裝備, **Then** 該裝備擁有 1 個主詞條和 4 個副詞條
2. **Given** 一件普通品質裝備, **When** 生成該裝備, **Then** 該裝備擁有 1 個主詞條和 0 個副詞條
3. **Given** 詞條池定義了各詞條的權重, **When** 大量生成副詞條, **Then** 各詞條出現頻率接近其權重比例
4. **Given** 一件裝備的主詞條是「生命百分比」, **When** 裝備升級, **Then** 主詞條數值按等級增加

---

### User Story 4 - 背包管理 (Priority: P2)

玩家擁有一個背包來存放未穿戴的物品。背包有容量上限，玩家可以查看、整理背包中的物品。

**Why this priority**: 背包是物品存放的基礎容器，與穿戴系統配合使用。

**Independent Test**: 可以透過向背包添加物品直到滿載，驗證容量限制是否生效。

**Acceptance Scenarios**:

1. **Given** 背包未滿, **When** 玩家獲得一件物品, **Then** 物品成功加入背包
2. **Given** 背包已滿, **When** 玩家嘗試獲得新物品, **Then** 系統拒絕並提示背包已滿
3. **Given** 背包中有多件物品, **When** 玩家查看背包, **Then** 可以看到所有物品的基本資訊
4. **Given** 背包中有一件物品, **When** 玩家丟棄該物品, **Then** 物品從背包中移除

---

### User Story 5 - 套裝效果 (Priority: P3)

當玩家穿戴同一套裝的多件裝備時，會觸發套裝效果。2 件套和 4 件套分別觸發不同等級的效果。

**Why this priority**: 套裝效果增加策略深度，但需要先有完整的穿戴和詞條系統。

**Independent Test**: 可以透過穿戴不同數量的同套裝備，驗證套裝效果是否正確觸發和計算。

**Acceptance Scenarios**:

1. **Given** 玩家穿戴了 2 件同套裝備, **When** 計算角色屬性, **Then** 2 件套效果生效並加成到角色數值
2. **Given** 玩家穿戴了 4 件同套裝備, **When** 計算角色屬性, **Then** 2 件套和 4 件套效果都生效
3. **Given** 玩家穿戴了 2 件 A 套裝和 2 件 B 套裝, **When** 計算角色屬性, **Then** A 和 B 的 2 件套效果都生效
4. **Given** 玩家卸下一件套裝裝備使同套數量低於門檻, **When** 計算角色屬性, **Then** 對應的套裝效果消失

---

### User Story 6 - 詞條快速查詢 (Priority: P3)

系統可以使用 Bitmask 機制快速檢測裝備是否擁有特定詞條或詞條組合。

**Why this priority**: 這是效能優化功能，在基礎功能完成後增加。

**Independent Test**: 可以透過建立擁有特定詞條的裝備，然後執行各種 Bitmask 查詢來驗證。

**Acceptance Scenarios**:

1. **Given** 一件裝備擁有暴擊和攻擊詞條, **When** 查詢是否擁有暴擊詞條, **Then** 回傳 true
2. **Given** 一件裝備擁有暴擊和攻擊詞條, **When** 查詢是否同時擁有暴擊和防禦詞條, **Then** 回傳 false
3. **Given** 一件裝備擁有暴擊和攻擊詞條, **When** 查詢是否擁有暴擊或防禦其中之一, **Then** 回傳 true

---

### User Story 7 - 資料持久化 (Priority: P3)

系統可以從 JSON 載入物品模板和詞條池定義，也可以將物品實例序列化/反序列化以便存檔和讀檔。

**Why this priority**: 持久化是遊戲存檔的必要功能，但需要先有完整的物品資料結構。

**Independent Test**: 可以透過建立物品、序列化、反序列化後比對資料是否一致。

**Acceptance Scenarios**:

1. **Given** JSON 格式的物品模板檔案, **When** 系統載入, **Then** 所有模板正確解析並可用於生成物品
2. **Given** 一個物品實例, **When** 序列化為 JSON, **Then** JSON 包含所有物品屬性
3. **Given** 序列化後的 JSON, **When** 反序列化, **Then** 還原的物品與原物品完全相同

---

### Edge Cases

- 背包滿時卸下裝備：系統拒絕卸下並提示「背包已滿，請先清理空間」
- 角色等級不足時嘗試穿戴高等級裝備會發生什麼？（系統應拒絕並提示等級不足）
- 生成物品時模板不存在會發生什麼？（系統應回傳錯誤或 nil）
- 詞條池為空時嘗試生成副詞條會發生什麼？（系統應處理空池情況）
- 套裝定義中包含不存在的物品 ID 會發生什麼？（系統應忽略無效 ID）

## Requirements *(mandatory)*

### Functional Requirements

#### 物品基礎系統

- **FR-001**: 系統 MUST 支援 5 個裝備欄位：頭盔(helmet)、身體(body)、手套(gloves)、鞋子(boots)、腰帶(belt)
- **FR-002**: 系統 MUST 為每個物品實例生成全域唯一的 UUID
- **FR-003**: 系統 MUST 區分物品模板（定義種類）和物品實例（實際存在的物品）
- **FR-004**: 系統 MUST 支援 5 種稀有度：普通(common)、優良(uncommon)、稀有(rare)、史詩(epic)、傳說(legendary)

#### 數值系統

- **FR-005**: 系統 MUST 支援以下角色數值：攻擊力(attack)、防禦力(defense)、最大生命(maxHP)、最大魔力(maxMP)、暴擊率(critRate)、暴擊傷害(critDamage)、速度(speed)
- **FR-006**: 系統 MUST 按公式計算最終數值：`最終值 = (角色基礎值 + Σ固定加成) × (1 + Σ百分比加成)`
- **FR-007**: 系統 MUST 在裝備穿戴/卸下時即時更新角色數值

#### 詞條系統

- **FR-008**: 每件裝備 MUST 擁有一個主詞條，數值隨裝備等級成長
- **FR-009**: 副詞條數量 MUST 根據稀有度決定：普通 0 條、優良 1-2 條、稀有 2-3 條、史詩 3-4 條、傳說 4 條
- **FR-010**: 系統 MUST 根據詞條池權重隨機生成副詞條
- **FR-011**: 系統 MUST 使用 Bitmask 支援詞條的快速查詢和組合判斷

#### 裝備穿戴系統

- **FR-012**: 系統 MUST 驗證物品只能裝備到對應的欄位（如頭盔只能裝備到頭盔欄位）
- **FR-013**: 系統 MUST 驗證角色等級滿足裝備的等級需求
- **FR-014**: 系統 MUST 在更換裝備時自動將舊裝備放回背包

#### 背包系統

- **FR-015**: 背包 MUST 有容量上限
- **FR-016**: 系統 MUST 在背包滿時拒絕添加新物品
- **FR-017**: 使用者 MUST 能夠從背包中移除/丟棄物品

#### 套裝系統

- **FR-018**: 系統 MUST 支援套裝定義，包含所屬物品和套裝效果
- **FR-019**: 系統 MUST 在穿戴足夠數量同套裝備時自動觸發套裝效果
- **FR-020**: 系統 MUST 支援多種套裝效果類型：數值加成(stat_bonus)、隊伍增益(team_buff)、條件觸發(conditional)、元素反應(elemental_reaction)

#### 資料持久化

- **FR-021**: 系統 MUST 支援從 JSON 載入物品模板
- **FR-022**: 系統 MUST 支援從 JSON 載入詞條池定義
- **FR-023**: 系統 MUST 支援物品實例的序列化與反序列化

### Key Entities

- **ItemTemplate（物品模板）**: 定義物品種類的藍圖，包含模板 ID、名稱、描述、欄位類型、稀有度、等級需求、基礎數值、屬性列表、資源名稱
- **Item（物品實例）**: 實際存在的物品，包含實例 UUID、所屬模板 ID、當前等級、主詞條、副詞條列表、詞條 Bitmask
- **Affix（詞條）**: 裝備上的屬性加成，包含類型、數值、是否為百分比
- **AffixPool（詞條池）**: 定義各欄位可出現的主副詞條及其權重
- **EquipmentSet（套裝）**: 定義套裝包含的物品和不同件數的效果
- **Stats（數值）**: 角色的各項能力數值集合
- **Inventory（背包）**: 存放未穿戴物品的容器，有容量限制
- **EquipmentSlots（裝備欄）**: 角色的 5 個裝備欄位，每個欄位最多裝備一件對應類型的物品
- **Avatar（角色）**: 玩家控制的角色，擁有基礎數值、裝備欄、背包

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 玩家可以在 1 秒內完成裝備的穿戴或卸下操作
- **SC-002**: 系統可以在 100 毫秒內計算角色穿戴所有裝備後的總數值
- **SC-003**: 詞條隨機生成的分布在 1000 次測試中與權重比例誤差不超過 5%
- **SC-004**: Bitmask 詞條查詢可以在常數時間 O(1) 內完成
- **SC-005**: 100% 的物品實例 UUID 保證唯一性
- **SC-006**: 所有裝備欄位限制規則正確執行，無法將錯誤類型的裝備穿戴到不匹配的欄位
- **SC-007**: 套裝效果在穿戴/卸下裝備後 100 毫秒內正確更新
- **SC-008**: 物品序列化後再反序列化，資料完整性達到 100%
- **SC-009**: 每個功能模組都有對應的單元測試，測試先於實作完成（TDD）
- **SC-010**: 所有單元測試在每次任務完成時皆為綠燈狀態

## Scope

**In Scope:**
- 純 Model 層實作：資料結構、業務邏輯、單元測試
- 物品系統核心功能（生成、穿戴、背包、詞條、套裝）

**Out of Scope:**
- UI/View 層實作
- ViewModel 或狀態綁定
- 網路同步或多人遊戲邏輯

## Development Methodology

**TDD（測試驅動開發）:**
- 先寫測試，再寫實作
- 進行下一項任務之前，必須確保當前所有單元測試都通過
- Red → Green → Refactor 循環

**架構原則:**
- 保持乾淨架構（Clean Architecture）
- 由於是練習專案，可適度簡化結構（例如：省略不必要的抽象層）
- 優先確保可測試性和單一職責

## Assumptions

- 背包容量上限採用常見 RPG 設定（如 100 格），可配置
- 物品等級上限為 20 級，與規格中範例一致
- 主詞條成長公式採用線性成長（每級固定增量）
- UUID 採用標準 UUID v4 格式
- 角色等級系統已存在，本系統僅讀取角色等級進行裝備需求驗證
- 同一副詞條類型在同一裝備上只能出現一次

## Clarifications

### Session 2026-01-30

- Q: 百分比加成的計算方式？ → A: 百分比加成與基礎值分開計算後相加，公式為：`最終值 = (基礎值 + 固定加成) × (1 + Σ百分比加成)`
- Q: 背包滿時卸下裝備的處理方式？ → A: 拒絕卸下並提示「背包已滿，請先清理空間」
- Q: 實作範圍界定？ → A: 純 Model 層，僅實作資料結構和業務邏輯，不含 UI
- Q: 開發方法論？ → A: 採用 TDD，每項任務完成前須通過所有單元測試；保持乾淨架構但可適度簡化
