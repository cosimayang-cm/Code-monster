# Code Monster #6: Console Board Game 棋盤遊戲引擎

## 學習目標

透過設計一個「VC 當搖桿、Console 當畫面」的回合制棋盤遊戲引擎，學習：
- **Protocol-Oriented Programming**：四個遊戲共用框架，透過 Protocol 定義共通介面
- **State Machine**：遊戲狀態管理（等待 → 進行中 → 結束）
- **Clean Architecture**：Engine 層只用 Foundation、VC 層只用 UIKit，徹底分離

## 技術需求

- Swift 5.9+
- UIKit（VC 層）
- Foundation（Engine 層）
- **架構不指定**：學員自行選擇 TCA 或 Protocol+MVVM，需在方案中寫出選擇理由

---

## 核心設計限制

### Console Output

所有遊戲畫面透過 `print()` 輸出到 Xcode Console，使用文字 + emoji 呈現棋盤。

### VC = Input Only

UIKit ViewController **只負責接收使用者操作**（按鈕點擊），不顯示任何遊戲畫面。VC 是搖桿，Console 是螢幕。

### 畫面輸出在 Console

所有遊戲畫面都顯示在 Xcode Console 上，具體怎麼實作由學員自行設計。

### 回合制

每次使用者操作產生一次 console print，顯示最新棋盤狀態。

---

## Shared Game Framework

四個遊戲有共通概念：**棋盤**、**操作**、**判勝負**、**渲染**。

鼓勵設計可複用的框架，讓新增一款遊戲時只需實作具體邏輯，不需重寫整個流程。具體的 Protocol / Enum 名稱與簽名由學員自行設計。

## 題目一：Tic-Tac-Toe 井字棋

### 規則

- 3×3 棋盤
- 兩位玩家輪流下子（❌ vs ⭕）
- 空格才能下
- 橫、直、斜三連線者贏
- 棋盤滿且無人三連線為平手

### Console 顯示範例

```
🎮 Tic-Tac-Toe
──────────────
  1   2   3
┌───┬───┬───┐
│ ❌ │ ⭕ │   │ A
├───┼───┼───┤
│   │ ❌ │   │ B
├───┼───┼───┤
│ ⭕ │   │ ❌ │ C
└───┴───┴───┘

Player ❌'s turn
```

### AI

- **Hard 難度**：Minimax 演算法，完美對弈，永不輸
- Minimax 在井字棋的搜尋空間小，不需 pruning

### VC

- 3×3 的 `UIButton` grid
- 點擊按鈕 → 送出 Move → Engine 處理 → AI 回應 → `print(renderer.render())`

### 驗收條件

- [ ] 3×3 棋盤正確渲染到 Console
- [ ] 空格可下，已有子的格子不可下
- [ ] 橫、直、斜三連線正確判定勝利
- [ ] 棋盤滿無三連線判定平手
- [ ] Hard AI 使用 Minimax，先手或後手都不會輸
- [ ] VC 只有 UIButton，不顯示棋盤
- [ ] 每次操作後 Console 印出最新棋盤

## 題目二：Connect Four 四子棋

### 規則

- 7 欄 × 6 列棋盤
- 兩位玩家輪流選擇一欄投入棋子
- **重力掉落**：棋子落到該欄最底部的空位
- 橫、直、斜四連線者贏
- 棋盤滿且無人四連線為平手

### Console 顯示範例

```

├───┼───┼───┼───┼───┼───┼───┤
│   │   │   │   │   │   │   │ 4
├───┼───┼───┼───┼───┼───┼───┤
│   │   │ 🔴 │   │   │   │   │ 3
├───┼───┼───┼───┼───┼───┼───┤
```

- 點擊按鈕 → 棋子掉落到該欄最底空位

- [ ] 棋盤滿無四連線判定平手
- [ ] AI 使用 Alpha-Beta Pruning，能擋明顯的勝著
- [ ] VC 只有 7 個按鈕（每欄一個），不顯示棋盤

## 題目三：Reversi 黑白棋

### 規則

- 8×8 棋盤
- 初始狀態：中心 4 格交叉放置黑白各 2 子
- 下子時必須**至少翻轉 1 顆對手的子**（8 個方向皆檢查）
- 無子可下時**跳過該回合**
- 雙方都無子可下時遊戲結束
- 棋子多的一方獲勝

### Console 顯示範例

```
  ├───┼───┼───┼───┼───┼───┼───┼───┤

  ├───┼───┼───┼───┼───┼───┼───┼───┤
```

**特殊顯示**：用 `*` 標記當前玩家可下的位置，並顯示每個位置的翻轉數量。

### AI

- **位置權重矩陣**：角落最高分、角落旁最低分
- **Alpha-Beta Pruning**：搜尋深度 4-6 層
- **角落策略**：優先搶角、避開角落相鄰格

### VC

- 8×8 的 `UICollectionView`
- 點擊格子 → 下子 → 翻轉 → AI 回應

### 驗收條件

- [ ] 8×8 棋盤正確渲染到 Console
- [ ] 初始狀態中心 4 子正確放置
- [ ] 8 個方向翻轉邏輯正確
- [ ] 必須能翻轉至少 1 子才能下
- [ ] 無子可下時自動跳過回合
- [ ] 雙方都無子可下時正確結束遊戲
- [ ] 用 `*` 標記可下位置，顯示翻轉數量
- [ ] AI 使用權重矩陣 + Alpha-Beta，具備角落策略
- [ ] VC 使用 UICollectionView，不顯示棋盤

## 題目四：2048

### 規則

- 4×4 棋盤，**單人遊戲**（非對戰）
- 上下左右滑動，所有數字方塊往該方向移動
- 相同數字碰撞合併（2+2=4, 4+4=8, ...）
- 每次滑動後，隨機在一個空格生成 2（90% 機率）或 4（10% 機率）
- **勝利條件**：合成 2048（可選擇繼續玩）
- **失敗條件**：棋盤滿且四個方向都無法合併

### Console 顯示範例

```
🎮 2048
──────────────────────
┌──────┬──────┬──────┬──────┐
│      │    2 │      │    2 │
├──────┼──────┼──────┼──────┤
│    4 │   16 │    8 │    4 │
├──────┼──────┼──────┼──────┤
│    2 │  128 │   64 │   32 │
├──────┼──────┼──────┼──────┤
│    4 │  256 │  128 │    8 │
└──────┴──────┴──────┴──────┘

Score: 4,892
Swipe: ⬆ ⬇ ⬅ ➡
```

**顯示要求**：數字靠右對齊，保持格子寬度一致。

### 核心演算法：Matrix Slide & Merge

四個方向的滑動合併邏輯：
1. 將該方向的每一行/列取出
2. 移除空格、壓縮
3. 相鄰相同數字合併（每格每次只合併一次）
4. 再次壓縮、補空格

### VC

- 四個方向按鈕：⬆ ⬇ ⬅ ➡
- 點擊方向按鈕 → 滑動合併 → 生成新方塊 → `print(renderer.render())`

### 驗收條件

- [ ] 4×4 棋盤正確渲染到 Console，數字靠右對齊
- [ ] 四個方向滑動合併邏輯正確
- [ ] 每格每次滑動只合併一次（例：2,2,2,2 → 4,4 而非 8）
- [ ] 滑動後隨機空格生成 2（90%）或 4（10%）
- [ ] 無法移動的方向不產生效果（不生成新方塊）
- [ ] 合成 2048 時顯示勝利訊息（可繼續玩）
- [ ] 棋盤滿且無法合併時顯示失敗訊息
- [ ] 正確計算並顯示分數（合併產生的數字累加）
- [ ] VC 只有四個方向按鈕，不顯示棋盤

## 測試策略

### Renderer 測試

驗證 console 輸出的棋盤畫面是否正確。

### Move 驗證測試

- Valid move：空格下子、正確欄位投入
- Invalid move：已有子的格子、已滿的欄、不能翻轉的位置、無效方向

### Win Detection 測試

- Tic-Tac-Toe：橫（3 種）、直（3 種）、斜（2 種）、平手
- Connect Four：橫、直、斜（左上→右下、右上→左下）、平手
- Reversi：雙方都不能下時結束、計算子數多寡
- 2048：合成 2048 勝利、棋盤滿且無法合併失敗

### AI 測試

- Tic-Tac-Toe Hard AI 永不輸（模擬多局）
- Connect Four AI 能擋明顯的勝著（設定即將四連線的局面）
- Reversi AI 具備角落策略（有角落可下時優先選角落）

### Reversi 特有測試

- 翻轉計數正確（每個方向分別驗證）
- 跳過回合（一方無子可下時自動跳過）
- 遊戲結束判定（雙方都不能下）

### 2048 特有測試

- 滑動合併正確（`[2,2,2,2]` → `[4,4,0,0]`、`[2,0,2,0]` → `[4,0,0,0]`）
- 無效滑動不生成新方塊
- 分數計算正確

### 整合測試

模擬完整遊戲流程：開始 → 多次操作 → 結束，驗證狀態轉換正確。

## 提示

1. **在 Game 1 建立完整 framework**：Game 2 / 3 / 4 只需新增 conformance
2. **Board / Move 用 struct**：值語義方便 Minimax 遞迴時複製棋盤
3. **畫面渲染與邏輯分離**：渲染只負責呈現，不包含遊戲邏輯
4. **State Machine 清楚定義**：遊戲在任何時刻都處於明確的狀態（等待中 / 進行中 / 結束），避免非法狀態轉換
